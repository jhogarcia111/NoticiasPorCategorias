# **Requisitos para la aplicación de Noticias para LinkedIn**

### Objetivo General
Desarrollar una aplicación que permita programar publicaciones en LinkedIn con imágenes generadas por IA y resúmenes de noticias de diferentes categorías. La app recopilará noticias diariamente, generará resúmenes concisos con IA si es necesario (para estandarizar longitud), creará imágenes personalizadas por noticia, y permitirá a usuarios seleccionar 6-8 noticias por post para un feed curado y atractivo.

## 1. Documento de Requisitos del Sistema

### Funcionalidades Principales

#### Roles de Usuario
- **Administrador**:
  - Gestionar todas las tablas del sistema
  - Configurar categorías de noticias y fuentes/portales externos (ej. activar/desactivar NewsAPI, RSS feeds)
  - Gestionar usuarios y permisos
  - Ver reportes de publicaciones (incluyendo métricas como likes/shares via LinkedIn API Insights)
  - Forzar recolección de noticias manualmente
  - Monitorear dashboard con KPIs, gráficas de uso/errores y tabla de portales para gestión operativa
- **Usuario Registrado**:
  - Conectar perfiles de LinkedIn (múltiples perfiles por usuario)
  - Seleccionar categorías de interés
  - Programar publicaciones (con templates editables para resúmenes e imágenes)
  - Seleccionar noticias manualmente o aleatoriamente (6-8 noticias) (con preview de post compilado)
  - Ver historial de publicaciones (con estado y métricas post-publicación)
  - Acceder a dashboard inicial con overview personal (posts pendientes, noticias sugeridas)

#### Gestión de Contenido
- Recolección automática diaria de noticias por categorías (usando NewsAPI como primaria, fallback a RSS; limitar a 50 noticias/día por categoría para costos)
- Generación de resúmenes con IA (ej. OpenAI GPT-4o-mini) para estandarizar a 100-200 palabras, enfocados en insights profesionales
- Generación de imágenes con IA para las noticias (prompts automáticos basados en título/resumen, usando Stability AI o DALL-E; resolución 1200x627 para LinkedIn)
- Programación de publicaciones en LinkedIn (con manejo de queue para posts masivos)
- Selección de noticias para publicar (compilar en un post único: intro + resúmenes + imágenes)
- Gestión de portales/fuentes: Admins pueden pausar fuentes para control de costos o compliance (ej. si un RSS falla frecuentemente).

#### Características Técnicas
- Frontend en React con estilo moderno (usar Tailwind CSS + shadcn/ui para componentes responsive)
- Backend con Supabase (PostgreSQL)
- Integración con LinkedIn API (v2, scopes: r_liteprofile, w_member_social, w_organization_social para posts)
- Integración con APIs de IA para generación de imágenes y resúmenes (OpenAI/Stability AI; fallback si quota excedida)
- Conexión con fuentes de noticias (RSS, NewsAPI, scraping) (evitar scraping heavy para compliance; usar NewsAPI v2)
- Manejo de errores: Retry para publicaciones fallidas (3 intentos), logs en tabla audit.
- Internacionalización: Soporte inicial para español/inglés.
- Performance: Paginación en listas de noticias/posts; cache con React Query.
- Dashboards: Inicial para todos los roles, con visualizaciones data-driven (Recharts para admin).

**Requisitos No Funcionales:**
- Seguridad: Encriptar tokens LinkedIn con Supabase Vault o pg_crypto.
- Escalabilidad: Supabase auto-escala; limitar users iniciales a 100.
- Accesibilidad: WCAG 2.1 AA.
- Testing: Unit tests con Jest, E2E con Cypress.

## 2. Esquema de Base de Datos - Supabase (PostgreSQL)

Mejoras: Agregué campos para IA (resumen_ai, ai_image_url), índices, tabla de audits, y encriptación básica para tokens. Usar extensiones como pg_trgm para búsquedas en noticias.

```sql
-- Habilitar extensiones útiles
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS pgcrypto;  -- Para encriptación

-- Tabla de usuarios (extensión de auth.users de Supabase)
CREATE TABLE profiles (
  id UUID REFERENCES auth.users(id) PRIMARY KEY,
  username TEXT UNIQUE,
  avatar_url TEXT,
  role TEXT CHECK (role IN ('admin', 'user')) DEFAULT 'user',
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
CREATE INDEX idx_profiles_role ON profiles(role);

-- Tabla de categorías de noticias
CREATE TABLE categories (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT,
  rss_feed_url TEXT,
  newsapi_category TEXT,  -- Ej. 'technology', 'business' para NewsAPI
  provider_type TEXT DEFAULT 'newsapi',  -- Ej. 'newsapi', 'rss', 'custom'
  api_key_encrypted TEXT,  -- Encriptado para creds de portales
  usage_count INTEGER DEFAULT 0,  -- Calls por día para monitoreo
  is_active BOOLEAN DEFAULT TRUE,
  created_by UUID REFERENCES profiles(id),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
CREATE INDEX idx_categories_active ON categories(is_active);
CREATE INDEX idx_categories_provider ON categories(provider_type);

-- Tabla de perfiles de LinkedIn
CREATE TABLE linkedin_profiles (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id UUID REFERENCES profiles(id) NOT NULL,
  profile_name TEXT NOT NULL,
  linkedin_id TEXT NOT NULL,
  access_token TEXT NOT NULL,  -- Encriptar en producción: pgp_sym_encrypt(access_token, 'key')
  refresh_token TEXT,  -- Encriptar también
  token_expires_at TIMESTAMP WITH TIME ZONE,
  profile_type TEXT CHECK (profile_type IN ('personal', 'company')) DEFAULT 'personal',
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
CREATE INDEX idx_linkedin_user ON linkedin_profiles(user_id);

-- Tabla de relación perfiles-categorías
CREATE TABLE profile_categories (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  profile_id BIGINT REFERENCES linkedin_profiles(id) ON DELETE CASCADE,
  category_id BIGINT REFERENCES categories(id) ON DELETE CASCADE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  UNIQUE(profile_id, category_id)
);

-- Tabla de noticias
CREATE TABLE news (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  category_id BIGINT REFERENCES categories(id) NOT NULL,
  title TEXT NOT NULL,
  summary TEXT,  -- Resumen original de fuente
  ai_summary TEXT,  -- Resumen generado por IA (estandarizado)
  content TEXT,
  source_url TEXT NOT NULL,
  image_url TEXT,  -- Imagen original de fuente
  ai_image_url TEXT,  -- Imagen generada por IA, almacenada en Supabase Storage
  source_name TEXT,
  published_at TIMESTAMP WITH TIME ZONE,
  is_processed BOOLEAN DEFAULT FALSE,  -- Flag para IA completada
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
CREATE INDEX idx_news_category ON news(category_id);
CREATE INDEX idx_news_processed ON news(is_processed);
CREATE INDEX idx_news_published ON news(published_at DESC);  -- Para noticias recientes

-- Tabla de publicaciones programadas
CREATE TABLE scheduled_posts (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  profile_id BIGINT REFERENCES linkedin_profiles(id) NOT NULL,
  post_content TEXT NOT NULL,  -- Compilación: intro + resúmenes + calls to action
  image_url TEXT,  -- Imagen principal (de una noticia o generada)
  scheduled_time TIMESTAMP WITH TIME ZONE NOT NULL,
  status TEXT CHECK (status IN ('pending', 'published', 'failed', 'retrying')) DEFAULT 'pending',
  posted_at TIMESTAMP WITH TIME ZONE,
  linkedin_post_id TEXT,  -- ID del post en LinkedIn para tracking
  error_message TEXT,  -- Para fallos
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
CREATE INDEX idx_scheduled_time ON scheduled_posts(scheduled_time);
CREATE INDEX idx_scheduled_status ON scheduled_posts(status);

-- Tabla de relación publicaciones-noticias
CREATE TABLE post_news (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  post_id BIGINT REFERENCES scheduled_posts(id) ON DELETE CASCADE,
  news_id BIGINT REFERENCES news(id) ON DELETE CASCADE,
  display_order INTEGER DEFAULT 0,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Nueva: Tabla de audits para logs
CREATE TABLE audits (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id UUID REFERENCES profiles(id),
  action TEXT NOT NULL,  -- Ej. 'post_published', 'news_fetched'
  details JSONB,  -- Datos flexibles
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Habilitar Row Level Security (RLS)
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE categories ENABLE ROW LEVEL SECURITY;
ALTER TABLE linkedin_profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE profile_categories ENABLE ROW LEVEL SECURITY;
ALTER TABLE news ENABLE ROW LEVEL SECURITY;
ALTER TABLE scheduled_posts ENABLE ROW LEVEL SECURITY;
ALTER TABLE post_news ENABLE ROW LEVEL SECURITY;
ALTER TABLE audits ENABLE ROW LEVEL SECURITY;

-- Políticas de seguridad para profiles
CREATE POLICY "Users can view their own profile" ON profiles
  FOR SELECT USING (auth.uid() = id);
CREATE POLICY "Users can update their own profile" ON profiles
  FOR UPDATE USING (auth.uid() = id);
CREATE POLICY "Admins can view all profiles" ON profiles
  FOR SELECT USING (
    EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'admin')
  );

-- Políticas para categories (solo admins)
CREATE POLICY "Admins manage categories" ON categories
  FOR ALL USING (
    EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'admin')
  );

-- Políticas para linkedin_profiles
CREATE POLICY "Users can view their own LinkedIn profiles" ON linkedin_profiles
  FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "Users can manage their own LinkedIn profiles" ON linkedin_profiles
  FOR ALL USING (auth.uid() = user_id);

-- Políticas para audits
CREATE POLICY "Users can insert audits" ON audits FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY "Admins view all audits" ON audits FOR SELECT USING (EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'admin'));

-- Más políticas RLS según necesidad...
```

### Endpoints Personalizados (usando Edge Functions de Supabase)
Mejora: Agrego payloads de ejemplo en JSON y manejo de errores (códigos HTTP 4xx/5xx). Usar Deno para Edge Functions.

#### Gestión de Perfiles de LinkedIn
```
POST /api/linkedin-profiles
// Payload: { profile_name: "Mi Perfil", profile_type: "personal" }
// Respuesta: { id: 1, access_token: "..." } (después de OAuth)

GET /api/linkedin-profiles?user_id=uuid
// Respuesta: [{ id: 1, profile_name: "...", is_active: true }]
```

#### Gestión de Categorías (solo admin)
```
GET /api/categories?active=true
// Respuesta: [{ id: 1, name: "Tecnología", newsapi_category: "technology" }]

POST /api/categories
// Payload: { name: "Tecnología", description: "...", newsapi_category: "technology" }

PUT /api/categories/:id { ... }
DELETE /api/categories/:id
```

#### Noticias
```
GET /api/news?category=:categoryId&limit=20&processed=true
// Respuesta: [{ id: 1, title: "...", ai_summary: "...", ai_image_url: "storage_url" }]

POST /api/news/refresh (solo admin - para forzar actualización)
// Trigger: Llama a NewsAPI, procesa con IA, marca is_processed=true
// Respuesta: { processed: 15, errors: 2 }
```

#### Programación de Publicaciones
```
GET /api/scheduled-posts?profile_id=1&status=pending
// Paginación: ?page=1&limit=10

POST /api/scheduled-posts
// Payload: { 
//   profile_id: 1, 
//   post_content: "Resumen compilado...",

// }

PUT /api/scheduled-posts/:id { status: "published", linkedin_post_id: "urn:li:..." }
DELETE /api/scheduled-posts/:id

POST /api/scheduled-posts/:id/publish-now  // Publica inmediatamente
// Respuesta: { success: true, linkedin_post_id: "..." } o error
```

#### Integración con LinkedIn

<a name="configuracion"></a>
## 4. Configuración de Supabase

### Pasos de configuración inicial
1. Crear cuenta en supabase.com
2. Crear nuevo proyecto (elige región cercana, ej. US East para latencia baja)
3. Configurar autenticación:
   - Habilitar registro por email y proveedores externos si futuro (Google)
   - Configurar URLs de redirección (ej. http://localhost:3000 para dev, https://tuapp.com para prod)
4. Ejecutar scripts SQL para crear tablas (usa SQL Editor en dashboard; prueba en staging project)
5. Configurar políticas RLS (verificar con tests en Supabase client)
6. Configurar Edge Functions necesarias (desplegar con Supabase CLI: supabase functions deploy)
7. Configurar almacenamiento para imágenes (bucket 'news-images' público para LinkedIn, privado para drafts)
8. Configurar cron para recolección: Usa Supabase pg_cron extension o externo (Vercel Cron invocando /api/news/refresh diariamente a las 6AM UTC)
9. Habilitar pg_cron para jobs diarios de stats: `SELECT cron.schedule('daily-stats', '0 0 * * *', 'UPDATE categories SET usage_count = (SELECT COUNT(*) FROM news WHERE category_id = id AND created_at > NOW() - INTERVAL ''1 day'');');`

### Variables de entorno necesarias
```env
REACT_APP_SUPABASE_URL=tu_url_de_supabase
REACT_APP_SUPABASE_ANON_KEY=tu_anon_key
REACT_APP_LINKEDIN_CLIENT_ID=tu_linkedin_client_id  # De developers.linkedin.com
REACT_APP_LINKEDIN_CLIENT_SECRET=tu_linkedin_client_secret
REACT_APP_NEWSAPI_KEY=tu_newsapi_key  # newsapi.org
REACT_APP_OPENAI_API_KEY=tu_openai_key  # Para resúmenes/imágenes (o Stability AI)
REACT_APP_LINKEDIN_REDIRECT_URI=https://tuapp.com/auth/linkedin/callback
SUPABASE_SERVICE_ROLE_KEY=tu_service_key  # Para Edge Functions admin
```

Costos Estimados Iniciales: Supabase free tier ok para <50 users; NewsAPI $49/mes (100k calls); OpenAI ~$0.01 por imagen/resumen (x50 noticias/día = ~$15/mes).

<a name="flujos"></a>
## 6. Flujos de Trabajo Principales

### Flujo de Autenticación
1. Usuario visita la aplicación
2. Si no está autenticado, redirigido a login/registro (con validación de email)
3. Tras autenticación exitosa, redirigido al dashboard (fetch perfil y perfiles LinkedIn)
4. Sesión mantenida con tokens JWT de Supabase (auto-refresh; logout limpia localStorage)

### Flujo de Conexión con LinkedIn
1. Usuario hace clic en "Conectar perfil de LinkedIn" (selecciona tipo: personal/company)
2. Redirección a OAuth de LinkedIn (state param para CSRF)
3. Usuario autoriza la aplicación (verifica scopes en dev portal)
4. Callback a nuestra app con código de autorización
5. Intercambio de código por token de acceso (y refresh_token; chequear expires_at en DB)
6. Almacenamiento seguro del token en Supabase (encriptado; test refresh cada 60 días)

### Flujo de Recolección de Noticias
1. Tarea programada (cron) se ejecuta diariamente (ej. 6AM UTC via pg_cron: SELECT cron.schedule('daily-news', '0 6 * * *', 'CALL refresh_news()');)
2. Para cada categoría activa, consulta fuentes de noticias (NewsAPI: GET /v2/everything?q=category&apiKey=...; fallback RSS con rss-parser en Edge Function)
3. Procesa y almacena nuevas noticias en la base de datos (dedup por title/source_url)
4. Genera resúmenes con IA si summary original >200 palabras o vacío
5. Genera imágenes con IA para cada noticia (prompt: "Ilustración profesional de [title] para LinkedIn, estilo moderno"; batch de 10 para eficiencia)
6. Actualiza base de datos con el contenido nuevo (set is_processed=true; log en audits)

### Flujo de Programación de Publicaciones
1. Usuario selecciona perfil de LinkedIn y categorías (fetch noticias no usadas recientemente)
2. Sistema muestra noticias disponibles de esas categorías (con previews AI)
3. Usuario selecciona manualmente 6-8 noticias o elige aleatorio (auto-compila post_content: "Descubre estas noticias clave: 1. [ai_summary1] [imagen1] ...")
4. Usuario programa fecha/hora de publicación (validar no overlap; min 1 hora futuro)
5. Sistema crea entrada en scheduled_posts y post_news
6. En el momento programado, Edge Function publica en LinkedIn (usa /api/linkedin/share; si falla, set status='failed' y retry en 1h)
7. Actualiza estado de la publicación (fetch metrics post-publicación via LinkedIn API; notificar admin por email si >3 fallos)

### Flujo de Manejo de Errores
- Si publicación falla (ej. token expired): Auto-refresh token, retry 3x.
- Logs en audits; dashboard muestra errores accionables.

### Flujos del Administrador
Estos flujos se activan en el AdminDashboard inicial, con navegación a secciones detalladas. Requieren rol 'admin' validado en API.

#### Flujo de Monitoreo y Dashboard Inicial
1. Admin inicia sesión y es redirigido al AdminDashboard.
2. Dashboard carga KPIs en paralelo (total users, posts status, error rate) via /api/admin/stats.
3. Visualiza gráficas: 
   - Posts por día (línea total + barras por usuario top).
   - Errores (barra por tipo, filtrable por fecha).
   - Uso app (usuarios activos, sesiones de audits).
4. Si error_rate >5%, highlight con alerta (ej. toast: "Alta tasa de fallos – revisar tokens").
5. Admin puede drill-down: Click en gráfica -> modal con detalles (ej. lista de posts fallidos).

#### Flujo de Gestión de Usuarios y Permisos
1. Desde dashboard, clic en "Gestionar Usuarios".
2. Lista paginada de profiles (fetch via Supabase query con RLS admin).
3. Acciones: Cambiar rol (user->admin), banear (set inactive), ver stats per user (posts count).
4. Confirmación para cambios sensibles; log en audits.

#### Flujo de Configuración y Gestión de Categorías/Portales
1. Clic en "Gestionar Fuentes/Portales" en dashboard.
2. Muestra SourcesTable: Lista de categories con provider_type, active toggle, usage_count.
3. Para editar: Modal con form (nombre, RSS URL, NewsAPI category, API key – encriptar on-save).
4. Toggle active: PUT /api/admin/sources/:id -> Si inactive, pausa recolección cron para esa fuente.
5. Ver "Último Error": De audits (ej. "NewsAPI quota exceeded" -> decidir desactivar).
6. Botón "Refrescar Uso": POST /api/admin/sources/refresh-usage para update daily counts.
7. Si se desactiva un portal (ej. por costos altos), notificar users via email (integra SendGrid).

#### Flujo de Reportes y Manejo de Errores
1. En dashboard, sección "Errores" con gráfica; clic -> Lista detallada de audits (filter por status='failed').
2. Para post fallido: Ver detalles (error_message), retry manual (POST /publish-now) o debug (ej. refresh token).
3. Generar reporte: Export CSV de stats (posts por user/día) via Edge Function.
4. Monitoreo proactivo: Cron job envía email diario si posts_failed >10.

#### Flujo de Forzar Recolección y Mantenimiento
1. Clic "Forzar Recolección Noticias" en dashboard.
2. POST /api/news/refresh -> Procesa solo fuentes active; progress bar en UI.
3. Post-ejecución: Update dashboard KPIs y log en audits.

Integración con Otros Flujos: Admins ven todo; users solo su dashboard. Logs de admin actions en audits para auditoría.

## 7. Integraciones Externas y Deployment

### Integraciones
- **NewsAPI**: Regístrate en newsapi.org; usa endpoints /top-headlines y /everything. Limita queries a 100/día inicial.
- **LinkedIn API**: Crea app en developers.linkedin.com; whitelistea redirect URIs. Para posts: Usa /ugcPosts con multipart/form-data para texto+imagen.
- **IA para Resúmenes/Imágenes**: OpenAI (GPT para texto, DALL-E 3 para imgs) o Stability AI (más barato para imgs). Prompts example: "Resume esta noticia en español profesional para LinkedIn: [content]".
- **Almacenamiento**: Sube imgs a Supabase Storage; genera URLs públicas con signed tokens para LinkedIn.
- **Emails/Notificaciones**: Integra Supabase con SendGrid o Resend para alerts.
- **Analítica Adicional:** Opcional Google Analytics para track uso (no en DB); para métricas LinkedIn, usa /organizationPageStatistics si company pages.

### Deployment
- **Frontend**: Vercel/Netlify (auto-deploy desde GitHub; env vars seguras).
- **Backend**: Supabase maneja todo; deploy Edge Functions via CLI.
- **CI/CD**: GitHub Actions: Test -> Build -> Deploy. Script: `supabase db push` para schema changes.
- **Monitoreo**: Supabase Logs + Sentry para errors; Google Analytics para uso.
- **Próximos Pasos**: 1. Setup Supabase y DB. 2. Implementa auth + LinkedIn OAuth. 3. Agrega recolección noticias. 4. Test end-to-end.
